## 2. Model Selected: SARIMAX (Primary) & Prophet (Linux/Mac only)

### SARIMAX: Windows-Compatible Production Model ✅

**SARIMAX** (Seasonal AutoRegressive Integrated Moving Average with eXogenous variables) is the **recommended model for this Windows system**.

**Model Specification:**
```python
from statsmodels.tsa.statespace.sarimax import SARIMAX

model = SARIMAX(
    endog=y,  # Occupancy time series
    exog=X,   # External regressors [is_holiday, is_sandwich, is_weekend]
    order=(1, 0, 1),              # ARMA(1,1) - non-seasonal component
    seasonal_order=(1, 0, 1, 7),  # Seasonal ARMA(1,1) with 7-day period
    enforce_stationarity=False,
    enforce_invertibility=False
)
```

**Why SARIMAX?**

✅ **Windows compatible** - No C++ compilation required  
✅ **External regressors** - Like Prophet, supports calendar features  
✅ **Confidence intervals** - Provides prediction intervals  
✅ **Seasonal patterns** - Weekly seasonality (7-day period)  
✅ **Well-established** - Decades of academic/industry use  
✅ **Interpretable** - Clear AR, MA, seasonal components  

**Ideal for refuge occupancy because:**
- Weekly patterns explicitly modeled (seasonal_order with s=7)
- Chilean calendar effects as exogenous variables
- Prediction intervals for uncertainty quantification
- No installation headaches on Windows

**Comparison to Prophet:**

| Feature | SARIMAX | Prophet |
|---------|---------|---------|
| Windows support | ✅ Yes | ❌ No (requires cmdstan) |
| External regressors | ✅ Yes (exog) | ✅ Yes (add_regressor) |
| Confidence intervals | ✅ Yes (95% CI) | ✅ Yes (customizable) |
| Seasonality | ✅ Single period (e.g., weekly) | ✅ Multiple (yearly + weekly) |
| Setup time | ⚡ <1 min | ⏳ 1-2 hours (MinGW + cmdstan) |
| Training speed | ⚡ Fast (~10 sec) | ⏳ Slower (~30 sec) |
| Interpretability | ✅ Transparent components | ✅ Decomposable |

**SARIMAX Model Parameters Explained:**

- **order=(1, 0, 1)**: Non-seasonal ARMA(1,1)
  - p=1: Autoregressive order (yesterday's occupancy influences today)
  - d=0: No differencing (data already stationary)
  - q=1: Moving average order (yesterday's error influences today)

- **seasonal_order=(1, 0, 1, 7)**: Seasonal ARMA with weekly period
  - P=1: Seasonal AR (last week's Saturday influences this Saturday)
  - D=0: No seasonal differencing
  - Q=1: Seasonal MA
  - s=7: Weekly seasonality

- **exog**: External regressors (is_holiday, is_sandwich, is_weekend)

### Prophet: Alternative for Linux/Mac Systems

**Prophet is available for systems with C++ compilation tools.**

### Why Prophet?

**Strengths:**
- **Handles seasonality**: Automatic yearly and weekly seasonality detection
- **Supports regressors**: Can incorporate external features (holidays, weekends)
- **Uncertainty quantification**: Provides prediction intervals (yhat_lower, yhat_upper)
- **Robust to missing data**: Handles gaps in time series
- **Interpretable**: Decomposes forecast into trend + seasonality + holidays
- **Production-ready**: Developed and maintained by Facebook (Meta)

**Ideal for refuge occupancy because:**
- Weekly patterns (weekend vs weekday)
- Yearly seasonality (summer vs winter tourism)
- Chilean calendar effects (holidays, sandwich days)
- Binary occupancy can be modeled as probability

### Model Configuration:

```python
from prophet import Prophet

model = Prophet(
    yearly_seasonality=True,      # Capture summer/winter patterns
    weekly_seasonality=True,       # Capture weekend vs weekday
    daily_seasonality=False,       # Not relevant for daily data
    seasonality_mode='additive',   # Additive seasonality (not multiplicative)
    interval_width=0.95            # 95% confidence intervals
)

# Add external regressors
model.add_regressor('is_holiday')
model.add_regressor('is_sandwich')
model.add_regressor('is_weekend')
```

**Seasonality mode rationale:**  
Additive seasonality assumes seasonal effects are constant across time (e.g., weekends always +X% occupancy), whereas multiplicative assumes effects scale with trend. For binary occupancy, additive is more appropriate.

## 3. Regressors Used and Why

Prophet allows adding **external regressors** (covariates) to capture effects beyond seasonality.

### Regressors Included:

| Regressor | Type | Description | Expected Effect |
|-----------|------|-------------|----------------|
| `is_holiday` | Binary | Official Chilean public holidays | **Strong positive** - Holidays drive refuge occupancy (guaranteed time off) |
| `is_sandwich` | Binary | Sandwich days (bridges) | **Moderate positive** - Behavioral pattern, partial adoption |
| `is_weekend` | Binary | Saturday/Sunday | **Positive** - Weekends baseline higher than weekdays |

### Why These Three?

**is_holiday** - Strongest demand driver:
- Official holidays = nationwide closures
- Guaranteed paid time off
- Predictable surges in outdoor recreation
- Example: Fiestas Patrias (Sept 18-19) → peak occupancy

**is_sandwich** - Chilean behavioral pattern:
- Workers take vacation day between holiday and weekend
- Creates 3-4 day long weekends
- Not universal but widespread
- Example: Holiday on Friday + worker takes Thursday off = 4-day weekend

**is_weekend** - Baseline weekly cycle:
- Weekends naturally higher occupancy than weekdays
- Saturdays vs Sundays may differ (captured by weekly_seasonality)
- Interacts with holidays (holiday on weekend still elevated)

### Regressors NOT Included (Future Work):

- **Temperature**: Weather impact on refuge visits  
- **School holidays**: Family vacation periods  
- **Pricing**: Dynamic pricing effects  
- **Marketing campaigns**: Promotional activity  
- **Road conditions**: Access during winter

**Why 60 days?**
- **2 months ahead**: Sufficient for operational planning (booking management, staffing)
- **Confidence window**: Beyond 60 days, prediction intervals widen significantly
- **Calendar coverage**: Captures mix of weekdays, weekends, potential holidays
- **Booking lead time**: Most refuge bookings occur 2-4 weeks before arrival

**Considerations:**
- Chilean calendar data extends to 2040 → Can forecast years ahead if needed
- Occupancy patterns may change seasonally → Consider re-training quarterly
- Model uncertainty increases with forecast distance → Monitor yhat_upper - yhat_lower

## 5. Output Structure

### Forecast DataFrame Columns:

| Column | Type | Range | Description |
|--------|------|-------|-------------|
| `ds` | date | 2026-01-10 to 2026-03-10 | Forecast date (60 days from last historical date) |
| `yhat` | float | [0, 1] | Predicted occupancy probability |
| `yhat_lower` | float | [0, 1] | Lower bound of 95% prediction interval |
| `yhat_upper` | float | [0, 1] | Upper bound of 95% prediction interval |
| `is_holiday` | int | 0 or 1 | Future holiday indicator (from calendar) |
| `is_sandwich` | int | 0 or 1 | Future sandwich day indicator |
| `is_weekend` | int | 0 or 1 | Future weekend indicator (derived) |

### Interpretation:

**yhat** (Point Prediction):
- Probability refuge is occupied on that date
- Example: `yhat = 0.75` → 75% chance of being occupied
- Binary conversion: `occupied = 1 if yhat > 0.5 else 0`

**yhat_lower / yhat_upper** (Uncertainty):
- 95% confidence interval around prediction
- Wider intervals = more uncertainty
- Example: `yhat ± (yhat_upper - yhat_lower)/2`

**Regressors** (Context):
- Included for transparency and debugging
- Show which calendar features active on forecast dates
- Enable "what-if" analysis (e.g., impact if holiday moved)

```csv

2026-09-18,0.92,0.85,0.97,1,0,0
```

**Interpretation:**  
September 18, 2026 (Chilean Independence Day)  
- 92% predicted occupancy  
- 95% confidence: [85%, 97%]  
- Holiday active (is_holiday=1) → Strong positive effect  
- Not a weekend (Friday) but holiday overrides

## 7. Limitations of First Model

### Known Issues:

❌ **Prophet does not work on Windows**  
- Requires C++ compilation tools (mingw32-make, Visual Studio Build Tools)
- cmdstan installation fails on this system
- **Impact:** Function raises ImportError with helpful message
- **Mitigation:** Use Docker, Linux/Mac, or alternative model (see below)

❌ **No train/test split**  
- Model trained on full dataset → Cannot evaluate out-of-sample performance
- **Impact:** Unknown forecast accuracy
- **Mitigation:** Implement time series cross-validation (rolling window)

❌ **Binary occupancy simplification**  
- y=1 treats all occupied nights equally (ignores occupancy count if >1 booking per night)
- **Impact:** Undercounts demand on high-occupancy nights
- **Mitigation:** Model occupancy count (Poisson regression) instead of binary

❌ **No model persistence**  
- Trained model not saved to disk (`models/` folder empty)
- **Impact:** Must retrain for every forecast (slow, non-deterministic)
- **Mitigation:** Add `model.save()` and `model.load()` calls

❌ **No forecast evaluation metrics**  
- No MAE, RMSE, or accuracy calculation
- **Impact:** Cannot quantify model performance
- **Mitigation:** Implement evaluation.py with metrics

❌ **Fixed 60-day horizon**  
- Horizon hardcoded in function
- **Impact:** Inflexible for different use cases
- **Mitigation:** Parameterize horizon (already implemented but not tested)

❌ **No seasonal decomposition output**  
- Prophet generates trend/seasonal/holiday components but not saved
- **Impact:** Cannot visualize what model learned
- **Mitigation:** Save component plots or extract components DataFrame

❌ **Future regressors assumed available**  
- Calendar file must cover forecast period (currently 2022-2040)
- **Impact:** Cannot forecast beyond 2040 unless calendar updated
- **Mitigation:** Implement regressor extrapolation or manual calendar extension

❌ **No outlier handling**  
- Extreme occupancy patterns (e.g., pandemic lockdowns) not filtered
- **Impact:** Model learns anomalies as normal patterns
- **Mitigation:** Add anomaly detection preprocessing

❌ **Additive seasonality assumption**  
- Assumes seasonal effects constant across time
- **Impact:** May underfit if occupancy grows exponentially
- **Mitigation:** Test multiplicative seasonality mode

### Performance Benchmarks (Not Yet Measured):

- **Training time:** Unknown (estimated ~30 seconds for 1,366 days)
- **Forecast accuracy (MAE):** Unknown
- **Prediction interval coverage:** Unknown (should be ~95% ideally)
- **Occupancy rate prediction error:** Unknown

### Option 2: SARIMAX with Exogenous Variables

**Model:** `statsmodels.tsa.statespace.sarimax.SARIMAX`  
**Regressors:** is_holiday, is_sandwich, is_weekend

**Advantages:**
- ✅ Works on Windows
- ✅ Supports external regressors (like Prophet)
- ✅ Seasonal ARIMA components
- ✅ Well-established statistical model

**Limitations:**
- ❌ Requires careful parameter tuning (p, d, q, P, D, Q, s)
- ❌ Sensitive to non-stationarity
- ❌ Harder to interpret than Prophet

**Implementation:**
```python
from statsmodels.tsa.statespace.sarimax import SARIMAX

model = SARIMAX(
    endog=train_df['y'],
    exog=train_df[['is_holiday', 'is_sandwich', 'is_weekend']],
    order=(1, 0, 1),          # (p, d, q) for non-seasonal part
    seasonal_order=(1, 0, 1, 7),  # (P, D, Q, s) for weekly seasonality
    enforce_stationarity=False
)
fitted = model.fit(disp=False)
forecast = fitted.forecast(steps=60, exog=future_exog)
```
